Key security principle: Row Level Security (RLS)
Supabase's RLS means hosts can only read/write their own listings and bookings. Guests can only see their own bookings. This is enforced at the database level — not just in your app code — so it's bulletproof.

Phase 2 — Auth & Database
Supabase handles both. Users sign up as guests or hosts. Row Level Security protects all data.
1
Create Supabase project & database schema
Go to supabase.com → New Project. Then run this SQL in the Supabase SQL editor to create your core tables.
sqlsupabase/schema.sql
-- Users (extends Supabase auth.users)
CREATE TABLE profiles (
  id          uuid REFERENCES auth.users ON DELETE CASCADE,
  full_name   text,
  avatar_url  text,
  role        text CHECK (role IN ('guest', 'host', 'both')),
  is_superhost boolean DEFAULT false,
  stripe_account_id text,           -- For host payouts
  created_at  timestamptz DEFAULT now(),
  PRIMARY KEY (id)
);

-- Listings (Hotels + Private Stays)
CREATE TABLE listings (
  id           uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  host_id      uuid REFERENCES profiles(id) ON DELETE CASCADE,
  type         text CHECK (type IN ('hotel', 'private')),
  title        text NOT NULL,
  description  text,
  location     text,
  price_per_night numeric NOT NULL,
  max_guests   int,
  images       text[],
  amenities    text[],
  status       text CHECK (status IN ('live', 'draft', 'paused')) DEFAULT 'draft',
  created_at   timestamptz DEFAULT now()
);

-- Bookings
CREATE TABLE bookings (
  id             uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  listing_id     uuid REFERENCES listings(id),
  guest_id       uuid REFERENCES profiles(id),
  check_in       date NOT NULL,
  check_out      date NOT NULL,
  guests         int,
  total_price    numeric,
  status         text CHECK (status IN ('pending','confirmed','cancelled','completed')),
  stripe_payment_intent_id text,
  created_at     timestamptz DEFAULT now()
);

-- Row Level Security
ALTER TABLE listings ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Hosts manage own listings" ON listings
  FOR ALL USING (host_id = auth.uid());
CREATE POLICY "Public can view live listings" ON listings
  FOR SELECT USING (status = 'live');

ALTER TABLE bookings ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Guests see own bookings" ON bookings
  FOR SELECT USING (guest_id = auth.uid());
CREATE POLICY "Hosts see bookings for their listings" ON bookings
  FOR SELECT USING (
    listing_id IN (SELECT id FROM listings WHERE host_id = auth.uid())
  );

Set up Supabase client in Next.js

import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'

export function createClient() {
  const cookieStore = cookies()
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    { cookies: { getAll() { return cookieStore.getAll() } } }
  )
}

05
Phase 3 — Bookings & Stripe Payments
Stripe Checkout for guests. Stripe Connect for host payouts. Webhooks to confirm bookings in your DB.
⚠️
Use Stripe Connect — not just Stripe Checkout
Because you're a marketplace (money flows from guests → SnapReserve → hosts), you need Stripe Connect. This lets you collect a platform fee and transfer the remainder to each host's bank account automatically.
1
Create Stripe Checkout session
When a guest clicks "Book Now", call this API route to create a payment session with your platform fee split off.
typescriptsrc/app/api/stripe/checkout/route.ts
import { stripe } from '@/lib/stripe'
import { createClient } from '@/lib/supabase/server'

export async function POST(req: Request) {
  const { listingId, checkIn, checkOut, guests } = await req.json()
  const supabase = createClient()

  // Fetch listing + host's Stripe account
  const { data: listing } = await supabase
    .from('listings')
    .select('*, profiles(stripe_account_id)')
    .eq('id', listingId)
    .single()

  const nights = daysBetween(checkIn, checkOut)
  const subtotal = listing.price_per_night * nights * 100 // cents
  const platformFee = Math.round(subtotal * 0.032) // 3.2% fee

  const session = await stripe.checkout.sessions.create({
    mode: 'payment',
    line_items: [{
      price_data: {
        currency: 'usd',
        product_data: { name: listing.title },
        unit_amount: subtotal,
      },
      quantity: 1,
    }],
    payment_intent_data: {
      application_fee_amount: platformFee,
      transfer_data: {
        destination: listing.profiles.stripe_account_id,
      },
    },
    metadata: { listingId, checkIn, checkOut, guests },
    success_url: `${process.env.NEXT_PUBLIC_URL}/booking/success?session_id={CHECKOUT_SESSION_ID}`,
    cancel_url:  `${process.env.NEXT_PUBLIC_URL}/listing/${listingId}`,
  })

  return Response.json({ url: session.url })
}
2
Handle Stripe webhooks to confirm bookings
When payment succeeds, Stripe pings this endpoint. You write the confirmed booking to Supabase and send confirmation emails.
typescriptsrc/app/api/stripe/webhook/route.ts
export async function POST(req: Request) {
  const sig = req.headers.get('stripe-signature')!
  const event = stripe.webhooks.constructEvent(
    await req.text(),
    sig,
    process.env.STRIPE_WEBHOOK_SECRET!
  )

  if (event.type === 'checkout.session.completed') {
    const { listingId, checkIn, checkOut, guests } = event.data.object.metadata
    const supabase = createAdminClient() // service role — bypasses RLS

    await supabase.from('bookings').insert({
      listing_id: listingId, check_in: checkIn, check_out: checkOut,
      guests: Number(guests), status: 'confirmed',
      stripe_payment_intent_id: event.data.object.payment_intent
    })

    await resend.emails.send({
      from: '[email protected]',
      to: guestEmail,
      subject: 'Your booking is confirmed! 🎉',
      react: <BookingConfirmEmail {...details} />
    })
  }

  return Response.json({ received: true })
}

Host onboarding — Stripe Connect
When a new host signs up, redirect them to create a Stripe Express account so they can receive payouts directly to their bank.
typescriptsrc/app/api/stripe/connect/route.ts
// Called when host clicks "Connect Bank Account" in Settings
export async function POST() {
  const account = await stripe.accounts.create({ type: 'express' })

  // Save account ID to host's profile
  await supabase.from('profiles').update({
    stripe_account_id: account.id
  }).eq('id', userId)

  // Redirect host to Stripe's onboarding UI
  const link = await stripe.accountLinks.create({
    account: account.id,
    refresh_url: `${url}/dashboard/settings`,
    return_url:  `${url}/dashboard/settings?connected=true`,
    type: 'account_onboarding',
  })

  return Response.json({ url: link.url })
}

Phase 4 — Wire Up the Host Dashboard
Connect your existing admin HTML to live Supabase data. The heavy lifting is already done design-wise.
1
Protect the dashboard with middleware
This middleware redirects unauthenticated users to login before they can access any dashboard route.
typescriptsrc/middleware.ts
import { createServerClient } from '@supabase/ssr'
import { NextResponse } from 'next/server'

export async function middleware(request: Request) {
  const supabase = createServerClient(...)
  const { data: { user } } = await supabase.auth.getUser()

  if (!user && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(new URL('/login', request.url))
  }
}

export const config = { matcher: ['/dashboard/:path*'] }
2
Fetch real dashboard stats
Replace hardcoded numbers in your dashboard with live Supabase queries. Run these server-side for instant load.
typescriptsrc/app/(dashboard)/dashboard/page.tsx
export default async function DashboardPage() {
  const supabase = createClient()
  const { data: { user } } = await supabase.auth.getUser()

  // Fetch all host's data in parallel
  const [listings, bookings, reviews] = await Promise.all([
    supabase.from('listings').select('*').eq('host_id', user.id),
    supabase.from('bookings')
      .select('*, listings!inner(*), profiles(*)')
      .eq('listings.host_id', user.id)
      .order('created_at', { ascending: false }),
    supabase.from('reviews')
      .select('*')
      .eq('host_id', user.id),
  ])

  // Compute stats
  const totalRevenue = bookings.data
    ?.filter(b => b.status === 'confirmed')
    .reduce((sum, b) => sum + b.total_price, 0)

  return <DashboardClient
    listings={listings.data}
    bookings={bookings.data}
    totalRevenue={totalRevenue}
  />
}
3
Real-time booking notifications with Supabase Realtime
Show new bookings in the dashboard instantly — no refresh needed. One line of setup.
typescriptsrc/components/dashboard/RealtimeBookings.tsx
'use client'
import { useEffect } from 'react'
import { createClient } from '@/lib/supabase/client'

export function RealtimeBookings({ hostId, onNew }) {
  useEffect(() => {
    const supabase = createClient()
    const channel = supabase
      .channel('new-bookings')
      .on('postgres_changes', {
        event: 'INSERT',
        schema: 'public',
        table: 'bookings',
        filter: `listing_id=in.(${hostListingIds})`
      }, payload => onNew(payload.new))
      .subscribe()

    return () => supabase.removeChannel(channel)
  }, [])
}

🌐
Launching on .co — migrating to .com later
This is totally fine and a common path. When you're ready to switch, the migration is painless on Vercel:

Step 1 — Buy snapreserve.com and add it to your Vercel project (Project → Domains → Add).
Step 2 — Set snapreserve.com as your primary domain in Vercel.
Step 3 — Keep snapreserve.co in Vercel as a redirect (301) → snapreserve.com. Vercel handles this in one click.
Step 4 — Update NEXT_PUBLIC_URL env var to snapreserve.com in Vercel dashboard.
Step 5 — Update your Stripe webhook URL and Supabase redirect URLs to .com.

SEO note: A 301 redirect from .co → .com passes full link equity — you won't lose any Google ranking you've built up on the .co domain.
